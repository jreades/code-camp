First Checkpoint

After a whirlwind introduction to Python you‚Äôve made it to the first checkpoint. This isn‚Äôt about us checking up on you, but rather about you having a chance to check your own understanding of the foundational concepts covered so far. It is really easy to click through the coding examples, look at the answer quickly ‚Äòjust to check‚Äô, and then come away with the impression that you‚Äôve understood what you are doing: ‚ÄúYeah, yeah, I made a little mistake there but it‚Äôs fine because I was close enough‚Ä¶‚Äù

We know that‚Äôs the case because we‚Äôve done it before you.

The challenge with coding is that, as the code becomes more complex, so do the issues. And if you haven‚Äôt learned to be careful with the basics and to fix them yourself then fixing the more subtle ‚Äòmistakes‚Äô will be that much harder. And you don‚Äôt want this to be any harder than it alreay is!

Here‚Äôs where the ‚Äòlearning a language‚Äô analogy breaks down a bit: because a fluent Spanish or Chinese speaker will often be able to make sense of what you‚Äôre trying to say as a learner, whereas the computer will just throw up its (metaphorical) hands through an exception. So learning the basics to the point where you can quickly find the simple errors really is important.

You Will Make Mistakes

So everyone makes mistakes. We do it all the time because. Computers are essentially stupid and do exactly what we tell them, so even tiny mistakes (aVariable vs avariable, for example) can cause a piece of code or a program to fail to run.

Note

Fun fact: the origin of the term bug was an actual bug (a moth) caught up in the mechanical relay of an early computer.

Bugs today are rarely of the insectile variety, and we now distinguish between three general types of error:

  1. Syntax errors are when the ‚Äògrammar‚Äô rules of Python (known as the syntax) have been violated, making it impossible for the code to be parsed (i.e. understood by Python);
  2. Logic errors have good syntax but there is an error in the order of statements or how they relate to one another;
  3. Semantic errors make programs do something other than what the programmer intended. Actually, not all programmers would distinguish between #2 and #3, but we‚Äôll make a case for these being different beasts below.

Here‚Äôs a nice little video on this topic‚Ä¶

When you first learn to code, syntax errors are the most common type of error because you are still learning the ‚Äòrules‚Äô of the language: indentation, the difference between = and ==, the need for a colon (:) on the end of some types of code‚Ä¶ Python is fairly good at telling you what has caused a syntax error when it‚Äôs trivial and happens on a single line of code, but it will get quite a bit harder as your code becomes more complex!

The other types of errors are ones that you learn to spot over time through experience and developing skills. Learning to spot logic errors is one of those wonderful transferrable skills that you can also use in everyday life: by learning to think through challenges at a higher level and then address them iteratively you are likely to become both a more careful thinker (i.e. how should I go about solving this issue?), and a more flexible and resilient thinker (i.e. ok, so that didn‚Äôt work out as I expected, what should do I do now?).

A good example of a logic error might be, ‚Äòtake a drink from your water bottle, put it in your backpack, walk to the library, and then put the top back on the bottle.‚Äô Severance (2014, p.14)

Many sites don‚Äôt even discuss the concept of a semantic error or treat it as the equivalent of a logical error, but we think it‚Äôs useful to set a bar beyond ‚Äòbasic‚Äô logic errors: these arise from interactions between parts of a program, or between a programming language and the underlying computer/operating system. So the code can (and usually does) run successfully but the output is a long way from what you expected, or the failures that you‚Äôre seeing don‚Äôt make sense. We‚Äôd suggest that spotting semantic errors is a sign that you are well on your way to thinking computationally. In short, you understand why your code is triggering strange and unexpected behaviour even though it looks like it should do exactly what you intended.

3.3 Nested Conditions

Conditional statements can be nested within one another. That is, once one conditional statement has been evaluated others may subsequently be evaluated (or not) depending on the result of the initial conditional statement. The code below shows an example of this.

  • Result
  • Interactive
if x != y:                                 #line 1
    print("x is not equal to y")
    
    if(x > y):                             #line 4
        print("x is greater than y")
        
    else:                                  #line 7
        print("x is less than y")

else:
    print("insert conclusion here")
x is not equal to y
x is greater than y

Note how the indentation makes it easier to work out which ‚Äòlevel‚Äô the code is operating on. In the code above, lines 4 and 7 are at the same indentation meaning that both will be skipped if the initial condition (on line 1) is False.
